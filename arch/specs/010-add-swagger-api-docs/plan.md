# Plan: Add Swagger/OpenAPI for REST Endpoints

## Goal

Add OpenAPI 3.0 spec to the Go backend so we can generate a Python client with `openapi-generator`.

REST endpoints only ‚Äî skip WebSocket `/ai` and `/ai/models`.

## Tech Stack

- **Tool**: [swaggo/swag](https://github.com/swaggo/swag) ‚Äî generates OpenAPI from Go annotations
- **UI**: [swaggo/http-swagger](https://github.com/swaggo/http-swagger) ‚Äî serves Swagger UI via Chi
- **Codegen**: `openapi-python-client` or `openapi-generator-cli` (future, not part of this plan)

### Why swaggo/swag

- Mature, widely used with Chi
- Annotations live next to handler code ‚Äî stays in sync
- Generates `swagger.json` that any codegen tool can consume
- No need to maintain a separate YAML file

## Approach

### 1. Install swaggo/swag

```bash
go install github.com/swaggo/swag/cmd/swag@latest
go get github.com/swaggo/http-swagger/v2
go get github.com/swaggo/swag
```

### 2. Add general API info to main.go

Add top-level swaggo annotations before the `main()` function:

```go
// @title Dimaist API
// @version 1.0
// @description Task management REST API
// @host localhost:3000
// @BasePath /
```

### 3. Add swagger route in main.go

Mount the Swagger UI handler on `/swagger/*`:

```go
import httpSwagger "github.com/swaggo/http-swagger/v2"
import _ "dimaist/docs"

r.Get("/swagger/*", httpSwagger.WrapHandler)
```

### 4. Annotate each REST handler

Each handler gets swaggo comment annotations. What to annotate:

| File | Handlers | Count |
|------|----------|-------|
| tasks.go | listTasks, getTask, createTask, updateTask, deleteTask, completeTask | 6 |

| sync.go | syncData | 1 |

```go
// @Summary List all tasks
// @Tags tasks
// @Produce json
// @Success 200 {array} database.Task
// @Failure 500 {string} string
// @Router /tasks [get]
func listTasks(w http.ResponseWriter, r *http.Request) {
```

### 5. Handle custom types for codegen clarity

`FlexibleTime` serializes as an ISO 8601 string. Add a swaggo type override so the generated spec uses `string` with `format: date-time` instead of an opaque object:

```go
// @Description ISO 8601 datetime string (e.g. "2026-01-15T09:30Z")
// swaggertype string
```

Or use `swag init --overridesFile` to map `utils.FlexibleTime` -> `string(date-time)`.

Similarly, `pq.StringArray` should map to `array of string` and `TimeArray` to `array of string(date-time)`.

### 6. Generate docs

```bash
cd backend && swag init
```

This creates `docs/` with `swagger.json`, `swagger.yaml`, and `docs.go`.

Regenerate after any annotation change.

### 7. Verify

- Start server, open `http://localhost:3000/swagger/index.html`
- Confirm all 14 endpoints appear with correct request/response schemas
- Download `swagger.json` and test with `openapi-generator validate`

## Response structs to define for Swagger

Some handlers return ad-hoc maps. Create small response structs so swaggo can generate clean schemas:

```go
type CreateTaskResponse struct {
    Task    database.Task `json:"task"`
    Warning string        `json:"warning,omitempty"`
}

type UpdateTaskResponse struct {
    Warning string `json:"warning,omitempty"`
}
```

`SyncResponse`, `FindResponse`, and `SearchResult` already exist as named structs ‚Äî they work as-is.

## Risks

- **Custom type mapping**: `FlexibleTime` and `TimeArray` may generate ugly schemas if not overridden. Mitigation: use swaggo's `swaggertype` directive or override file.
- **Ad-hoc response maps**: `createTask` returns `map[string]any`. Mitigation: create named response structs (see above).
- **Keeping docs in sync**: Annotations can drift from actual behavior. Mitigation: add `swag init` to CI or a pre-commit check.
